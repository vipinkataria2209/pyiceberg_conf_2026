The Write Path — Deep Dive

Everything inside table.append(), partitioned append, and overwrite() — from DataFrame to atomic S3 commit.

🗺️ The Big Picture

One call to table.append(batch) triggers a precise 6-stage pipeline. Stages 1–2 are pure in-memory Python. Stages 3–5 write 3 files to S3. Stage 6 is the atomic HTTP gate that makes it all visible.

🧩 Key Components

Orchestrator
Transaction
table/__init__.py

Top-level coordinator. Created implicitly by table.append(). Accumulates TableUpdate and TableRequirement objects and drives the commit. Nothing hits the network until commit_transaction().

Snapshot Producer
_FastAppendFiles
table/update/snapshot.py

"Fast" because it reuses existing manifests from the parent snapshot — only writes a new manifest for new files, never a full rewrite. Collects DataFile objects via append_data_file().

Parquet Writer
write_parquet()
io/pyarrow.py

Converts the PyArrow Table to Parquet and streams it to S3 via PyArrowFileIO. Also computes per-column min/max/null stats and packages them into a DataFile.

Data File Descriptor
DataFile
manifest.py

Represents one physical Parquet file. Contains S3 path, record count, file size, partition value, and column-level statistics used for file pruning at read time.

Metadata Index Entry
ManifestEntry
manifest.py

Wraps a DataFile with: status (ADDED / EXISTING / DELETED), snapshot_id, sequence_number, and file_sequence_number.

Optimistic Lock
AssertRefSnapshotId
table/update/__init__.py

Sent with every commit — "only accept this if the current snapshot is still X." Two racing writers: exactly one wins; the other gets CommitFailedException.

⚙️ The 6 Stages in Detail

1
In Memory — No Network

Transaction Opens

table.append(batch) creates a Transaction and initializes a _FastAppendFiles snapshot producer. Pure Python object creation — nothing written anywhere yet.

# Internally triggered by table.append(batch)
tx = Transaction(table)
fast_append = tx._fast_append_files()  # _FastAppendFiles init
2
In Memory — No Network

Schema Validation

_check_pyarrow_schema_compatible() confirms every Arrow column maps to a valid Iceberg field. Type mismatches fail fast — before any data is written to S3.

# Arrow dtype  →  Iceberg type
string         →  StringType()
timestamp[us]  →  TimestampType()
double         →  DoubleType()
# Mismatch → raises ValueError immediately, no S3 writes
3
S3 Write #1 — Parquet Data File

DataFrame → Parquet → S3

bin_pack_arrow_table() splits the table into write tasks. Each task goes to write_parquet() which streams to S3 and returns a DataFile with full column stats.

data/00000-0-550e8400-e29b-41d4-a716.parquet

DataFile Fields

4
S3 Write #2 — Manifest File (.avro)

Manifest File Written

The DataFile is wrapped in a ManifestEntry (status=ADDED) and written as Avro via ManifestWriterV2. This is the file index — future reads use it to skip files entirely.

metadata/uuid-m0.avro

status          = ADDED
snapshot_id     = 2985870393
sequence_number = 1
data_file       = DataFile(…)
💡
Why Avro? Manifest files use Avro because Iceberg's metadata schema is rich and nested — handles evolving schemas far better than JSON at scale.
5
S3 Write #3 — Manifest List + Snapshot

Manifest List & Snapshot Created

The manifest list (snap-*.avro) is the table of contents — it lists every manifest file. The Snapshot object is built in memory referencing it. Still not visible to readers.

metadata/snap-2985870393-1-uuid.avro

snapshot_id   = 2985870393
parent_id     = null     ← first append
operation     = "append"
summary       = {added-data-files: 1, added-records: 50}
⚠️
Not visible yet. All 3 files are on S3 — but no reader sees them until Stage 6 updates the catalog.
6
HTTP POST — Atomic Catalog Commit

Atomic Catalog Commit — The Gate

Single HTTP POST with AddSnapshotUpdate + AssertRefSnapshotId. The catalog validates atomically — accepts or rejects the entire commit.

POST /v1/namespaces/pyiceberg_demo/tables/sensor_readings/commit
{
  "requirements": [{ "type": "assert-ref-snapshot-id", "snapshot-id": null }],
  "updates":      [{ "action": "add-snapshot", … },
                   { "action": "set-snapshot-ref", "ref-name": "main" }]
}
✅
200 OK: snapshot is live — all readers now see 50 rows.
❌
409 Conflict: another writer committed first. PyIceberg raises CommitFailedException. Call table.refresh() and retry — PyIceberg does not auto-retry.

🔐 ACID Guarantees

A
Atomicity

The catalog commit is a single HTTP POST — all or nothing. If it fails, S3 files exist but are invisible (orphans).

C
Consistency

AssertRefSnapshotId ensures the table state at commit matches what the writer read. Schema requirements validate field IDs are unchanged.

I
Isolation

Readers always scan a consistent snapshot. A writer mid-flight is invisible — readers see old or new, never in-between.

D
Durability

Once the catalog confirms, the snapshot is durable. Metadata JSON on S3 is the source of truth — survives restarts.

🗂️ Partitioned Append — The Big Picture

Partitioning adds one extra step before Stage 3: bin_pack_arrow_table() groups rows by partition value and generates one Parquet file per distinct value. Everything else — manifest, snapshot, catalog commit — is identical to a plain append.

💡
Key insight: The partition spec lives in the table metadata, not in the Parquet files. Iceberg knows where data lives without opening a single file — partition pruning is 100% metadata-driven.

🧩 Partition-Specific Components

Partition Definition
PartitionSpec
partitioning.py

Defines which fields to partition on and which transform to apply. Stored permanently in table metadata. The field_id=1000 is the spec's own ID — separate from the source field ID.

PartitionSpec(
  PartitionField(
    source_id=1,          # sensor_id field
    field_id=1000,
    transform=IdentityTransform(),
    name="sensor_id"
  )
)
Row Binner
bin_pack_arrow_table()
io/pyarrow.py

Groups rows of the PyArrow Table by their partition value before any file is written. Each group becomes a separate WriteTask. For 150 rows across 5 sensor IDs → 5 write tasks → 5 Parquet files.

Path Strategy
LocationProvider
table/locations.py

Generates the S3 path for each partition file. The default SimpleLocationProvider creates {data_path}/{partition_path}/{filename}, producing the familiar sensor_id=PICARRO-001/ folder structure.

Read-time Pruner
_ManifestEvalVisitor
expressions/

At read time, checks each file's recorded partition_value in the manifest. A query WHERE sensor_id = 'PICARRO-003' opens exactly 1 file out of 5 — without reading any Parquet at all.

⚙️ Write Path Stages (Partitioned)

1
In Memory — No Network

Transaction + Schema Validation

Same as plain append — Transaction opens, _FastAppendFiles initialises, schema is validated. No network.

★
Extra Step — Partition Binning

bin_pack_arrow_table() — Row Grouping

This step does not exist in a plain append. The 150-row table is scanned and split into groups by sensor_id value — producing 5 WriteTask objects, one per distinct PICARRO ID.

all_batches (150 rows, 5 sensor_ids)
  → WriteTask(sensor_id=PICARRO-001, ~30 rows)
  → WriteTask(sensor_id=PICARRO-002, ~30 rows)
  → WriteTask(sensor_id=PICARRO-003, ~30 rows)
  → WriteTask(sensor_id=PICARRO-004, ~30 rows)
  → WriteTask(sensor_id=PICARRO-005, ~30 rows)
3
S3 Write #1–5 — 5 Parquet Files

5 Parquet Files Written in Parallel

Each WriteTask becomes a separate write_parquet() call. Files land in partition-specific S3 folders. Each returns its own DataFile with the partition_value field set.

data/sensor_id=PICARRO-001/00000-0-uuid1.parquet
data/sensor_id=PICARRO-002/00000-0-uuid2.parquet
data/sensor_id=PICARRO-003/00000-0-uuid3.parquet
data/sensor_id=PICARRO-004/00000-0-uuid4.parquet
data/sensor_id=PICARRO-005/00000-0-uuid5.parquet
⚠️
Folder names are a side effect. The partition metadata lives in the manifest — not the folder name. Iceberg does not rely on the path to determine partition value.
4
S3 Write #6 — Manifest File (5 Entries)

One Manifest — 5 Entries

All 5 DataFile objects are wrapped in ManifestEntry records (status=ADDED) and written to a single Avro manifest. Compare: plain append had 1 entry per manifest.

metadata/uuid-m0.avro  →  5 ManifestEntry records
  entry[0]: DataFile(PICARRO-001, partition_value={sensor_id:PICARRO-001})
  entry[1]: DataFile(PICARRO-002, partition_value={sensor_id:PICARRO-002})
  …
5
S3 Write #7 — Manifest List

Manifest List + Snapshot

Same as plain append — one snap-*.avro pointing to the single manifest. The snapshot summary now reports added-data-files: 5.

6
HTTP POST — Atomic Catalog Commit

Same Atomic Gate

Identical to plain append — AddSnapshotUpdate + AssertRefSnapshotId. Concurrency control works exactly the same way regardless of partitioning.

✅
Result: 5 Parquet files committed in a single atomic snapshot. Any reader sees all 5 files or none.

⚡ Read-time Benefit — Partition Pruning

The real payoff of partitioning shows up at read time. Here's what happens for a filtered scan:

# Query: only PICARRO-003 rows
table.scan(row_filter=EqualTo("sensor_id", "PICARRO-003")).to_arrow()

# What PyIceberg does:
1. Load manifest list (snap-*.avro)
2. For each ManifestEntry, check partition_value vs filter
   → PICARRO-001: SKIP  (partition_value != PICARRO-003)
   → PICARRO-002: SKIP
   → PICARRO-003: READ  ✅ open this Parquet file
   → PICARRO-004: SKIP
   → PICARRO-005: SKIP
# Result: 1 file opened out of 5 — without touching any Parquet

🔄 Overwrite — The Big Picture

table.overwrite(df, overwrite_filter=…) replaces rows matching a filter with new data. Unlike a database UPDATE, there is no in-place mutation. Old files stay on S3 untouched — only the snapshot changes which files are "current".

💡
Append vs Overwrite: Append only adds ADDED entries. Overwrite adds both ADDED (new file) and DELETED (old file) entries in the same manifest — committed atomically.

🧩 Overwrite-Specific Concepts

Operation
overwrite()
table/__init__.py

Takes a replacement DataFrame and an overwrite_filter expression. First scans the current snapshot to find affected files, then writes the replacement, then commits both ADDED and DELETED entries atomically.

Manifest Status
DELETED Entry
manifest.py

An overwrite manifest contains ManifestEntry(status=DELETED) for old files. This entry tells readers "this file existed in an older snapshot but is no longer current." The physical file stays on S3.

Cleanup
expire_snapshots()
table/maintenance.py

Old files referenced only by expired snapshots can eventually be deleted using table.expire_snapshots(). Until then, time travel to the old snapshot still works — the original PICARRO-001 rows are still readable.

Filter Evaluation
EqualTo("sensor_id", …)
expressions/

The overwrite filter is used in two ways: first to find which existing files to delete (manifest scan), then to filter rows in the new DataFrame for validation. Only files whose partition value matches are touched.

⚙️ Overwrite Write Path Stages

1
In Memory

Transaction Opens

Same as append — Transaction created, snapshot producer initialized. Schema validated against replacement DataFrame.

★
Extra Step — Scan for Affected Files

Find Files to Delete

This step does not exist in a plain append. PyIceberg scans the current snapshot's manifest to find every DataFile whose partition_value matches the overwrite_filter. For EqualTo("sensor_id","PICARRO-001") — only 1 of the 5 partition files is identified.

# Manifest scan (metadata-only — no Parquet opened)
manifest entries checked: 5
  PICARRO-001 → MATCH → schedule for DELETED entry
  PICARRO-002 → skip
  PICARRO-003 → skip
  PICARRO-004 → skip
  PICARRO-005 → skip
⚡
Partition pruning works here too. Only the matching partition file is identified — the other 4 Parquet files are never opened.
3
S3 Write #1 — New Parquet File

Write Replacement Data

Identical to Stage 3 of a plain append — the replacement DataFrame (20 PICARRO-001 rows) is written as a new Parquet file. A new DataFile is returned with updated stats (elevated co2_ppm min/max).

data/sensor_id=PICARRO-001/00000-0-uuid-new.parquet
← 20 rows, co2_ppm range: 420.0–429.5
4
S3 Write #2 — Manifest with ADDED + DELETED

Two-Entry Manifest

The overwrite manifest contains both kinds of entries in one Avro file — the new file as ADDED and the old file as DELETED. This is what makes the swap atomic.

metadata/uuid-m0.avro:
  ManifestEntry(status=DELETED, data_file=old-PICARRO-001.parquet)
  ManifestEntry(status=ADDED,   data_file=new-PICARRO-001.parquet)

# Other partition files are carried over from the parent snapshot's manifests
# _FastAppendFiles._existing_manifests() reuses them — no rewrite
5
S3 Write #3 — Manifest List

Snapshot Created

The manifest list (snap-*.avro) points to: the new overwrite manifest (with ADDED+DELETED) + the 4 carried-over manifests from the parent snapshot for PICARRO-002 through PICARRO-005.

summary: {
  operation:          "overwrite",
  added-data-files:   "1",   ← new PICARRO-001 file
  deleted-data-files: "1",   ← old PICARRO-001 file
  added-records:      "20",
  deleted-records:    "~30"  ← original row count varied
}
6
HTTP POST — Atomic Catalog Commit

Same Atomic Gate

Identical to append — AssertRefSnapshotId ensures no other writer modified the table since the scan in Step ★. If the table changed between the scan and the commit, the commit is rejected and you must retry.

⚠️
TOCTOU window: There's a gap between when files-to-delete are identified (★) and when the commit fires (6). If another writer appends PICARRO-001 rows in between, the commit is rejected — the AssertRefSnapshotId catches it.
